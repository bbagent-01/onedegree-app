## 7. Stale defaults & shape drift

### 7.1 `access_settings` column DEFAULT is stale

**Live column default** (from information_schema):

```json
{
  "message":            { "type": "min_score", "threshold": 10 },
  "see_full":           { "type": "min_score", "threshold": 10 },
  "see_preview":        { "type": "anyone" },
  "request_book":       { "type": "min_score", "threshold": 20 },
  "request_intro":      { "type": "anyone" },
  "view_host_profile":  { "type": "anyone" }
}
```

This is the pre-019 **6-key shape**. Migrations 019 + 020 collapsed it to the **4-key shape** in every existing row but **did not change the column DEFAULT**:

```json
{
  "see_preview":           { "type": "anyone" },
  "full_listing_contact":  { "...": "..." },
  "allow_intro_requests":  true,
  "preview_content":       { "show_title": true, "show_price_range": true, "...": "..." }
}
```

**Impact**: any future `INSERT INTO listings(...)` that omits `access_settings` will land with the legacy 6-key shape. The trust normalizer in `src/lib/trust/types.ts` apparently tolerates both, but new rows would diverge from the seed/migrated shape. Easy fix in mig 045: `ALTER COLUMN access_settings SET DEFAULT '{...new shape...}'::jsonb`.

### 7.2 Other JSONB column defaults — clean
- `preview_photos` default = NULL (column is orphan anyway).
- `cancellation_policy_override` default = NULL (intentional — fall-through to host default).
- `specific_user_ids` default = `'{}'::uuid[]` (legacy column; will go away).

### 7.3 `property_type` CHECK vs UI mismatch

UI labels: House · Apartment · Condo · Townhouse · Cabin · Other. DB CHECK: only `apartment`/`house`/`room`/`other`. `propertyTypeToDb()` collapses Condo→apartment and Townhouse/Cabin→house, and folds the place-kind into `room` for Private/Shared. Detail is preserved only inside `meta.propertyLabel` — which means filter/sort by "Cabin" against the real column is impossible.

### 7.4 `default_availability_status` CHECK vs wizard mismatch

Wizard collects `available` / `unavailable` / `possibly`. Column CHECK: `available` / `possibly_available` / `blocked`. The wizard never sends this, so the mismatch never reaches the DB — but if we wire up the write (recommended), the value mapping needs to be `possibly`→`possibly_available` and `unavailable`→`blocked`.

### 8.4 Backfill from `description` meta-blob into new columns

Because every existing listing's bedrooms/beds/etc lives inside `<!--meta:{...}-->`, mig 045 should:

1. Use a PL/pgSQL block to extract the meta JSON from `description` (regex `^<!--meta:(\{.*?\})-->`).
2. Populate the new columns from the parsed JSON (`max_guests`, `bedrooms`, `beds`, `bathrooms`, address parts, `weekly_discount_pct`, `monthly_discount_pct`, `extended_overview`, etc.).
3. Strip the meta-comment from `description` so reads no longer need `parseListingMeta`.

This is the most-error-prone step. Recommend writing it in TypeScript in a one-off seed-style script that reads + writes via the service role rather than as raw PL/pgSQL — easier to test and to bail out on parse failures.

### 8.5 Wizard / API rewires (not part of mig 045 itself, but the next-session work)

Once mig 045 is in:
- `POST /api/listings` body must accept `place_kind`, `property_label`, `max_guests`, `bedrooms`, `beds`, `bathrooms`, address parts, `cleaning_fee`, `weekly_discount_pct`, `monthly_discount_pct`, `default_availability_status`, the four extended-description fields, and `preview_settings` (if split out).
- `PATCH /api/listings/[id]` whitelist needs the same additions.
- `src/lib/listing-meta.ts` becomes a read-only fallback for legacy rows (or is deleted entirely after backfill verification).
- `src/lib/listing-data.ts` function exports (`getListingsForViewer`, `getListingForViewer`) and the four legacy-column types should be deleted.

## 9. Open questions for Loren

1. **`cleaning_fee` write path.** The column exists and is read in 6+ places (booking flows include it in totals). The wizard pretends to collect it but only stuffs it in the meta-blob. Existing rows therefore have NULL `cleaning_fee` and a `meta.cleaningFee`. **Decision: in mig 045 backfill, copy `meta.cleaningFee → cleaning_fee` and rewire the wizard to write the column directly?** (Recommend yes.)

2. **`default_availability_status` value set.** Wizard offers Available / Unavailable / Possibly. Column CHECK is `available` / `possibly_available` / `blocked`. **Decision: rename wizard values to match column (preferred), or change the column CHECK to match the wizard?** (Recommend renaming wizard.)

3. **Reserved word risk: `state`.** I proposed `region` to avoid PostgreSQL/ORM friction. Some teams keep `state` and quote it. **Pick: `state` or `region`?** Either works, just be consistent.

4. **Preview toggles — split out or leave in `access_settings`?**
   - **A (split)**: new `preview_settings jsonb` column. Cleaner separation: `access_settings` = WHO, `preview_settings` = WHAT. Easier RLS reasoning later.
   - **B (leave)**: keep them inside `access_settings.preview_content`. No migration churn; current code works.
   - Recommend A, but it's a judgment call.

5. **`min_trust_gate` — drop, or wire up?**
   - **A (drop)**: yank the column + delete the 5 readers in the same migration PR. The column has been at 0 for every row for the entire history of mig 013 — the gate-display code is dead.
   - **B (defer)**: keep until a future "trust-gate as a feature" session decides whether to revive it. Adds debt; column stays at 0.
   - Recommend A.

6. **Place type → `property_type` collapse.** The 4-value CHECK forces Condo/Townhouse/Cabin into apartment/house/house. If we add `property_label` (6-value), do we want to **also widen `property_type`** to match the UI, or **drop `property_type` entirely** in favor of `property_label`? Affects browse filtering.

7. **Drop legacy access columns in 045 or defer to a cleanup-only mig?** `preview_visibility`, `full_visibility`, `min_trust_score`, `specific_user_ids` are all legacy text-enum + listing-data.ts. Bundling them into 045 means coordinating a `listing-data.ts` deletion in the same PR. Splitting them out keeps 045 additive-only. Recommend bundle (less ceremony at alpha scale).

8. **`address` text column — drop unconditionally?** Zero readers + zero writers found. Confidence high, but flagging to confirm before droppage.

9. **House rules: keep as `text`, or normalize to a structured blob?** The wizard collects 4 booleans (no_smoking, no_pets, no_parties, quiet_hours) + custom text and joins them with `\n`. Browse-side filtering on "no pets" is impossible against a text blob. Out of scope for 045 (would touch UI), but worth deciding before wiring up filters.
